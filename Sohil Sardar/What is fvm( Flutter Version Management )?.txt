          # PERFECT ANSWER (For An InterView)
FVM (Flutter Version Management) is a command line tool that allow developers
to manage multiple flutter SDK version per project, rather than relying on the 
single global installed version on the machine.

The core problem it solves : different project always required different Flutter
version. Without FVM, you constantly run 'flutter upgrade' or 'flutter downgrade'
globally, which break other projects.

FVM creates per project SDK crashes, pins specific flutter versions to individuals
projects and seamlessly switches between them when you navigate between project 
folders.

Think of it like 'nvm' for node.js, 'rvm' for ruby, or 'pyenv' for python
- but specifically for flutter.

          # CORE FEATURES OF FVM:
i. Per - project SDK isolation:
   ---   ------- --- ---------- Each project has its own Flutter SDK cache.
ii. '.fvmrc' configuration:
     ------  -------------- JSON file storing the flutter version for 
                            flutter project.
iii. Symbolic link management:
     -------- ---- ---------- FVM creates symlinks to the cached SDK versions.
iv. Global fallback:
    ------ --------- If no per project version is set, 
                     FVM fallback to global flutter.
v. Release Channel Support:
   ------- ------- -------- Works with stable, beta, dev, master channel.
vi. Cache optimization:
    ----- ------------- Download sdk once per version, shares across projects
                        needing same version.

        # INSTALLATION & SETUP:
1. Install FVM: CODE: # Via pub (global activation)
   ------- ---  ----  dart pub global activate fvm
# Or via Homebrew (macOS)   brew install fvm
# Verify installation     fvm --version

2. Configure Your Flutter Project:
   --------- ---- ------- ------- # Navigate to your Flutter project
                                  cd my_flutter_app
# Pin a specific Flutter version to this project
fvm use 3.22.0 --force
# Or pin to a channel
fvm use stable

3. What FVM Creates:
   ---- --- --------
my_flutter_app/
├── .fvm/
│   ├── flutter_sdk -> /Users/user/fvm/versions/3.22.0  (symlink)
│   ├── fvm_config.json  (or .fvmrc)
│   └── version
├── .fvmrc  (project configuration file)
└── (rest of your project)

4. .fvmrc File Example
{
  "flutterSdkVersion": "3.22.0",
  "flavors": {
    "dev": "3.19.0",
    "prod": "3.22.0"
  }
}

          # COMMON INTERVIEW FOLLOW UP QUESTIONS:
Q) Why not just use flutter version like React Native?
ans) Flutter doesn't have a built in version mangement command.
The official recommendation is to use FVM for team project.
Without it, developers must manually manage the SDK versions
through environement vairable or homebrew, which is an error prone.

Q) Does FVM work with flavors/environments?
ans) yes, FVM support flavors. You can have different flutter versions
for dev, staging and production in the same project using the 'flavor'
key in '.fvmrc'.
CODE SET: {
  "flutterSdkVersion": "3.22.0",
  "flavors": {
    "dev": "3.19.0",
    "staging": "3.21.0",
    "prod": "3.22.0"
  } }

Q) How does FVM affect IDE performance?
ans) FVM creates 'symlinks', so IDE treat it like a normal flutter SDK.
There's virtually no performance overhead. Many team commit the 
'.fvm/flutter_sdk' symlink to '.gitignore' but 
commit .fvmrc to share version constraints.

Q) Can FVM manage multiple projects with the same version?
ans) Yes - FVM cashes each version once globally. 
Ten projects using Flutter 3.22.0 will all symlink 
to the same cached SDK, saving disk space.

Q) What's in your .gitignore for FVM?
# FVM
.fvm/flutter_sdk
.fvm/version   # Keep this file
!.fvmrc

            🎬 Summary (The "Elevator Pitch")
FVM is a version manager for Flutter that lets you pin different 
SDK versions to different projects. Instead of having one global Flutter 
version that breaks when projects diverge, FVM caches SDKs centrally and symlinks 
them per project using a .fvmrc config file. It's essential for team projects, 
CI/CD pipelines, and developers working on multiple Flutter apps with different 
version requirements. Think of it as nvm for Flutter — you might not need it for 
a single personal project, but any serious team should adopt it from day one.
