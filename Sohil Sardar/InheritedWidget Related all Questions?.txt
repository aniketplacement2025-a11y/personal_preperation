                      CORE UNDERSTANDING QUESTIONS?
 1. What is Inherited Widget?
 ans) InheritedWidget is a special flutter widget used to efficiently share data down 
 the widget tree without manually passing data through constructors.
 It helps multiple widgets access common data like:   *Theme
 *User data           *Localization          *Authentication state.
 without prop drilling.

 2. Why does Flutter Provide InheritedWidget?
 ans) Flutter provide Inherited Widget to solve the problem of passing shared data 
 deeply to multiple widget levels.       Without It:
 Home( theme: theme, child: Screen( theme: theme, child: AnotherScreen( theme: theme, ), ), ),
 this becomes messy and Inherited Widget helps to avoids this.

 3. What problem does it solve?
 ans) It solves prop drilling and enables efficient shared state access across the
 widget tree. Main Problem solved:
 *Reduces constructor passing      *Centralized Shared data       *Efficient rebuild handling
 *Better Scalability.

 4. How is it different from passing props via constructors?
 ans) Constructor Passing: *Manual *Repetative *Hard in deep trees.
 Inherited Widget: *Centralized sharing *Cleaner architecture 
 *Widgets are directly access shared data.
 Best Interview Line: Constructor passing works for small trees, But 
 InheritedWidget is better for app - wide sharing data.

                   WIDGET TREE QUESTIONS:
 1. How does InheritedWidget share data across the widget tree?
 ans) InheritedWidget places data high in a widget tree, and descendent widget access it using 
 BuildContext.              Example CODE:  MyInheritedWidget.of(context)
 Flutter searches Upward for the current context until it finds the nearest matching 
 InheritedWidget.

 2. Can parent widgets access child InheritedWidgets?
 ans) No Because
 "Inherited Widget only shares data downward to descendent widgets".
 Because flutter searches upward in the widget tree.
 Parents above the widget tree.

 3. How does flutter finds the nearest widget?
 ans) Flutter traverses upward through the element tree using BuildContext until 
 it finds the nearest matching Inherited widget type. Internally CODE:
        context.dependOnInheritedWidgetOfExactType<MyWidget>()

 4. What happens internally when u call Theme.of(context)?
 ans) Theme.of(context) internally uses 'dependOnInheritedWidgetOfExactType' to search 
 Upward for the nearest theme inheritedWidget. Flutter :
 1. Start from current context.             2. Moves Upward.
 3. FInd nearest Theme.                 4. Registers Dependency.
 4. Rebuild widget if Theme changes.

                  DEEP INTERNAL QUESTIONS:
 1. what does 'updateShouldNotify()' do?
 ans) 'updateShouldNotify()' tells flutter whether dependent widgets should rebuild
 when the InheritedWidget updates. Example:
 @override  bool updateShouldNotify(MyWidget oldWidget) {   return data != oldWidget.data;  }

 2. why is 'updateShouldNotify' important for performance?
 ans) It prevents unnecessary widget rebuilds.
 Flutter only rebuilds dependent widgets when data actually changes.
 This improves:
 *performance     *rendering efficiency     *UI responsiveness.

 3. What happens if it always returns true?
 ans) CODE: bool updateShouldNotify(oldWidget) => true;
 All dependent widgets rebuild every time, even if data didn’t change.
 Problem: *unnecessary rebuilds  *performance waste  *large UI cost.

 4. What happens if it always returns false?
 ans) CODE: bool updateShouldNotify(oldWidget) => false;
 Dependent widgets never rebuild even if the data changes.
 Problem: *stale UI  *incorrect data rebuilds  *UI not updated.
